> 编码前必须先产出设计文档（Spec）。本文件定义 Spec 模板和要素要求。

---

```

  → ai_service.chat_with_ai()

## 6. 错误处理矩阵
| 异常场景 | 处理方式 | 用户看到的 |
|----------|---------|-----------|
| ERP 401 | 返回登录提示 | "登录已过期，请重新登录" |
| LLM 429 | 切换备用模型 | 正常响应（可能有延迟） |
| 超长输入 | 拒绝请求 | "输入内容过长" |
| 工具调用超限 | 终止循环 | 已返回部分结果 |

## 7. 测试要点
- [ ] 正常场景：标准输入 → 正确输出
- [ ] 边界场景：空输入 / 超长输入 / 并发请求
- [ ] 异常场景：ERP 不可达 / 模型不可用 / 安全问题拦截
```

## 2. Spec 质量要求

### Grading 标准

| 维度 | 要求 |
|------|------|
| **完整性** | 7 个章节全部填写，不能有 "TODO" 或空内容 |
| **准确性** | 接口设计字段类型、错误码必须与实际代码一致 |
| **可验证性** | 测试要点必须具体，不可写 "测试一下" 这种模糊描述 |
| **简洁性** | 每章控制在半页以内，不堆砌无关内容 |

### 简化版 Spec（适用场景）

以下情况可用简化版（只需 1-3 章）：
- 单文件小改动（<50 行）
- 纯 Bugfix（不改变接口）
- 配置调整（环境变量、提示词等）

```markdown
# [Bugfix/小改动] 简短标题

## 改动
- 文件：xxx.py 第 N 行
- 原因：xxx
- 变更：xxx

## 影响
- 无接口变更 / 仅影响 xxx 场景
```

## 3. 设计方案评审要点

复杂功能（涉及 3+ 文件改动或 2+ 模块联动）的 Spec 需额外考虑：

1. **向后兼容**：旧客户端能否正常工作？
2. **性能影响**：新增调用是否增加延迟？是否需要缓存？
3. **可观测性**：新功能是否添加了足够的日志？
4. **复用性**：是否有机会抽取为通用工具/模块？

---

## 4. 设计原则

1. **渐进式迭代**：先核心路径，再边缘场景，不一次铺满
2. **可量化验证**：每个设计决策后面跟一个验证方法（怎么测？怎么知道它对了？）
3. **语言适配**：本项目 Python + TypeScript 双后端，设计时需标注是否影响对端
