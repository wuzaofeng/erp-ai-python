# WorkflowRules — 工作流规范

> 核心原则：**先设计、后编码**。杜绝"想到哪写到哪"。

---

## 开发流程（强制）

```
需求理解 → Spec 设计 → 编码实现 → 自测验证 → 审查 → 修复 → 完成汇报
```

### 1. 需求理解
- 读清楚需求描述，不明确处主动追问
- 确认涉及哪些模块、哪些接口、哪些数据表

### 2. Spec 设计（必须产出）
**编码前必须先产出 Spec 文档**，包含以下要素：

| 要素 | 说明 |
|------|------|
| **目标** | 要解决什么问题，一句话说清 |
| **影响范围** | 涉及哪些文件/模块/接口 |
| **接口设计** | 新增/修改的 API 端点、请求体/响应体 Schema |
| **数据流** | 请求从哪里来 → 经过哪些层 → 最终返回什么 |
| **错误处理** | 列出所有可能的异常情况及兜底策略 |
| **测试要点** | 至少列出 3 个测试场景（正常 + 边界 + 异常） |

### 3. 编码实现
- 严格按 Spec 实现，不随意扩展范围
- 遵循 [CodingStandardRules](./CodingStandardRules.md) 和 [SecurityRules](./SecurityRules.md)
- 每完成一个函数/方法，立即在代码内写清楚注释

### 4. 自测验证
- 参考 [UnitTestRules](./UnitTestRules.md) 编写测试用例
- 覆盖：正常输入 → 边界条件 → 异常输入 → 并发/性能（视情况）

### 5. 审查
- 自查一遍：类型注解完整？异常处理到位？安全规则符合？
- 复杂改动建议走一遍 `eval/run_eval.py` 回归测试

### 6. 修复 + 汇报
- 发现问题立即修复，并在工作完成汇报中标注修了什么
- 汇报格式：`做了什么 → 改了哪些文件 → 测试结果`

---

## 渐进式迭代原则

1. **最小可行交付**：先完成核心功能，次要优化后续迭代
2. **每步可验证**：每一步完成后都能跑起来验证
3. **不铺大摊子**：一次改动不超过 5 个核心文件，超出则拆分为多个 PR

---

## 禁止事项

- ❌ 无 Spec 直接写代码（除非是单行 bugfix）
- ❌ 改动超出需求范围（"顺便修一下"也不行）
- ❌ 不经测试就声称完成
- ❌ 在 PR/commit 中包含调试用的 print/console.log
